为什么要使用worker

  • 解决当前标签页处于非活跃状态时， 定时器(setTimeout, setInterval)计时不准确的问题。

    说起来，最开始准备写这篇博文的时候， 在chrome里通过上面的DEMO来进行测试， 结果发现web worker里的计时也不准确， 然而在番茄钟里面那个计时却是正确的， 因为这件事情， 就一直没有写下去， 后来再进行测试的时候， 就正确了， 真是玄乎。

  • 进行复杂运算， 防止用户界面无法响应（如上传文件的时候获取文件MD5）

几个要点

  • 同源策略
  • 数据的传递是通过拷贝而非共享

在webpack中使用

  • worker-loader
    这里需要注意的是， 虽然说在worker里引入其他脚本是使用importScripts()的方法， 但是import在里面依然可以使用， webpack会自动对其进行解析。

优化点

  • 缓存
    由于每次生成一个worker实例的时候， 都会对目标worker文件产生一次请求， 所以如果能够将对应的内容(也就是worker对应的js文件)缓存下来的话， 就可以避免重复请求。然后使用URL.createObjectURL的方法创建内容， 这个内容也是存在缓存里的内容。

  • 对于有多个地方业务需要使用到不同worker的情况， 可以封装一个函数作为代理， 对于worker的获取都通过这个函数

    如下， 通过worker-loader引入指定的worker

问题

  • 每次调用worker都应该生成一个实例， 而不是复用之前的worker， 当然， 具体情况还是要视业务而定。

  • worker进程也会存在延时的问题， 初步推测是浏览器版本的问题

    在下图中可以看到， 在浏览器标签页不再活跃的一段时间之后， window.setInterval就已经不再继续计时了, 而worker里的依然很坚挺。

浏览器版本：canary, Chrome/60.0.3082.3

  • 兼容性问题
    • 不同浏览器下worker的作用域里能够访问的全局变量存在不一致的情况， 比如在safari下是不能够访问File对象的
    • IE10支持worker, 但是当worker文件的地址是通过对象URL来生成的话， 会报错。

记一次事件捕获的妙用
